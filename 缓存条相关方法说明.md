# 缓存条相关方法说明

本文档整理播放器 UI 中「缓存条」所涉及的所有方法、数据流与切歌时的重置逻辑。

---

## 一、数据流概览

```
NLCacheManager.cacheProgressForURL:
        ↑（每 0.5 秒定时器 / 进入页面刷新时查询）
NLPlayerManager
        → cacheProgressSubject.sendNext(@(cacheProgress))
        ↓
NLMusicPlayerViewController 订阅 cacheProgressSignal
        → [self.playerView updateCacheProgress:cacheProgressNum.floatValue]
        ↓
NLMusicPlayerView.updateCacheProgress:
        → [self.cacheProgressView setProgress:progress animated:YES]
        ↓
UI：进度条下方一层 UIProgressView 显示 0.0～1.0 的缓存比例
```

---

## 二、各模块方法清单

### 1. NLCacheManager（数据源）

**文件**：`NLSpotify/Modules/播放器/NLCacheManager.h`、`NLCacheManager.mm`

| 方法 | 说明 |
|------|------|
| `- (float)cacheProgressForURL:(NSURL *)url` | 返回当前 URL 的缓存进度 `0.0 ~ 1.0`（已缓存字节 / 总长）。内部用 `totalLengthForURL:`、`cachedRangesForURL:`、`isFullyCachedForURL:`；已整首缓存返回 1，无 total 或无 ranges 返回 0。 |

**依赖的其它方法**（缓存条只读不写）：

- `totalLengthForURL:`：DB 中该 URL 的 totalLength  
- `cachedRangesForURL:`：DB 中已缓存区间（NSRange 数组）  
- `isFullyCachedForURL:`：是否已整首缓存（.mp3 或 isFinished）

---

### 2. NLPlayerManager（定时推送 + 切歌重置）

**文件**：`NLSpotify/Modules/播放器/NLPlayerManager.h`、`NLPlayerManager.m`

| 属性 / 方法 | 说明 |
|-------------|------|
| `@property (readonly) RACSignal<NSNumber *> *cacheProgressSignal` | 对外只读信号，缓存进度 0.0～1.0，与 `progressSignal` 同频（约 0.5 秒一次）。 |
| `cacheProgressSubject`（私有） | 内部 RACSubject，驱动 `cacheProgressSignal`。 |
| `- (void)addPeriodicTimeObserver` | 播放进度定时器（0.5 秒）。在回调里除发送 `progressSubject` 外，用 `currentSong.playURL` 调 `[NLCacheManager sharedManager] cacheProgressForURL:url]`，再 `[cacheProgressSubject sendNext:@(cacheProgress)]`。 |
| `- (void)playWithURL:(NSURL *)url` | 切歌时：`[cacheProgressSubject sendNext:@(0.f)]` 把缓存条置 0，再替换 item；不在此处移除/重加定时器。 |

---

### 3. NLMusicPlayerViewController（订阅并更新 View）

**文件**：`NLSpotify/Modules/播放器/播放器页面/NLMusicPlayerViewController.m`

| 位置 | 说明 |
|------|------|
| `bindPlayer` 里 `[manager.cacheProgressSignal subscribeNext:...]` | 订阅缓存进度，收到 `NSNumber` 后直接 `[self.playerView updateCacheProgress:progressNum.floatValue]`（不按 URL 过滤）。 |
| `songSignal` 订阅里 | 切歌时 `[self.playerView updateCacheProgress:0]`，与 PlayerManager 的 playWithURL 重置双重保险。 |
| `refreshUI` 无歌曲时 | `[self.playerView updateCacheProgress:0]`。 |
| `refreshUI` 有歌曲时 | `url = song.playURL`，`cacheProgress = url ? [[NLCacheManager sharedManager] cacheProgressForURL:url] : 0.f`，再 `[self.playerView updateCacheProgress:cacheProgress]`。 |

---

### 4. NLMusicPlayerView（UI 展示）

**文件**：`NLSpotify/Modules/播放器/播放器页面/NLMusicPlayerView.h`、`NLMusicPlayerView.m`

| 属性 / 方法 | 说明 |
|-------------|------|
| `@property (nonatomic, strong) UIProgressView *cacheProgressView` | 缓存条控件，叠在 UISlider 下方；`progress` 0～1 表示已缓存比例。 |
| `progressContainerView` | 容器，先放 `cacheProgressView`，再放 `progressSlider`，保证缓存条在底层。 |
| **创建**（setupUI） | `cacheProgressView.progressTintColor = [UIColor colorWithWhite:0.55 alpha:1]`（中间灰）；`trackTintColor = [UIColor tertiarySystemFillColor]`（未缓存段与未播一致）。 |
| **约束** | `[cacheProgressView mas_makeConstraints:... make.edges.equalTo(progressContainerView)]`，与 slider 同区域。 |
| `- (void)updateCacheProgress:(float)progress` | 对外接口；内部 `[self.cacheProgressView setProgress:progress animated:YES]`。 |

**头文件**（NLMusicPlayerView.h）：`- (void)updateCacheProgress:(float)cacheProgress;` 声明供 VC 调用。

## 三、切歌时缓存条「先满再缩」的排查方向（定时器方案已回退）

当前实现已回退：`cacheProgressSignal` 只发 `NSNumber`，VC 不做 URL 过滤；`playWithURL` 不再移除/重加定时器，只发一次 0。若现象仍出现，可从下面方向排查：

1. **主队列顺序**  
   定时器与 `playWithURL` 都在主队列：若切歌前定时器已入队，可能先执行「上一首 1.0」再执行 `playWithURL` 的 0，导致先满再缩。可加日志确认：在 VC 的 `updateCacheProgress:` 或订阅里打印 `progress` 与 `currentSong.playURL`，看是否先收到 1.0 再收到 0。

2. **refreshUI 调用时机**  
   `refreshUI` 在 **viewDidLoad** 里调用一次，会用 `currentSong` 查 `cacheProgressForURL` 并设缓存条。若 VC 在切歌瞬间才加载（例如从列表点进播放页），可能先执行了别处逻辑再执行 refreshUI，出现短暂错位。可确认是否有其它入口调用 `refreshUI` 或直接调 `updateCacheProgress:`。

3. **NLCacheManager 返回值**  
   `cacheProgressForURL:` 依赖 DB 的 `totalLengthForURL:`、`cachedRangesForURL:`、`isFullyCachedForURL:`。若新歌 URL 在切歌瞬间误用旧数据（例如同一 URL 复用、DB 未及时更新），可能一度返回 1.0。可在 `cacheProgressForURL:` 或调用处打日志，确认新歌首次查询是否异常。

4. **多处写缓存条**  
   会更新缓存条的有：`cacheProgressSignal` 订阅、`songSignal` 订阅（切歌时置 0）、`refreshUI`。三者执行顺序若与预期不符，也可能造成先满再缩。可统一在 `updateCacheProgress:` 内打日志，观察调用栈与先后顺序。

## 四、调用关系小结

| 调用方 | 被调方法 | 作用 |
|--------|----------|------|
| NLPlayerManager（playWithURL 开头） | `sendNext:@(0.f)` → 替换 item（不移除/重加定时器） | 切歌时发 0 |
| NLMusicPlayerViewController（cacheProgressSignal） | `updateCacheProgress:progressNum.floatValue` | 收到即更新，不按 URL 过滤 |
| NLPlayerManager（定时器） | `[NLCacheManager sharedManager] cacheProgressForURL:url]` | 取当前曲缓存比例 |
| NLPlayerManager（定时器） | `[cacheProgressSubject sendNext:@(cacheProgress)]` | 推给订阅者 |
| NLMusicPlayerViewController（songSignal） | `[self.playerView updateCacheProgress:0]` | 切歌时置 0 |
| NLMusicPlayerViewController（refreshUI） | `[NLCacheManager sharedManager] cacheProgressForURL:url]` 再 `updateCacheProgress:` | viewDidLoad 时同步一次 |
| NLMusicPlayerView | `[self.cacheProgressView setProgress:progress animated:YES]` | 刷新缓存条长度 |
